# 飞书skill · 笔记整理 + 画板可视化工作流

**CRITICAL — 开始前 MUST 先用 Read 工具读取 [`../lark-shared/SKILL.md`](../lark-shared/SKILL.md)（认证、权限、全局参数）。**
然后按需读取 [`../lark-doc/SKILL.md`](../lark-doc/SKILL.md)（读取/创建文档）与 [`../lark-whiteboard/SKILL.md`](../lark-whiteboard/SKILL.md)（画板创作）。

> **💸 成本提示(必看)**:本 skill 是 **token 重耗** 任务——要逐张读高分辨率课件图、正文又长。两条硬约束帮你省额度:
> 1. **建议单独开一个新会话跑本 skill**,跑完 `/clear`。别和别的任务挤在一个长会话里——长会话里读过的图每轮都会被重新计费,这是最大浪费源。
> 2. **整理类任务建议用 Sonnet 而非 Opus**(够用、单价低)。开工前先把需求一次问全,避免返工(返工=双倍消耗)。

## 适用场景

- "帮我整理这篇飞书笔记，能可视化的用画板画出来"
- "把这份文档归纳一下，另存一份新的，别动原文"
- "整理 + 画板可视化"
- "这篇笔记里有课件图片，先把图里的内容提取成文字再整理"
- "文档里嵌了截图/课件图，整理时把图都转成文字、表格或画板，别留静态图"

## 铁律

1. **原文档只读不改** —— 全程只对源文档用 `docs +fetch`，绝不 `docs +update` 源文档。
2. **整理结果另存为新文档** —— 用 `docs +create` 新建，默认存入个人云空间（my_library），**不要放进 wiki 知识库**，除非用户当次明确要求。
3. **可视化只用画板（whiteboard），绝不生成静态图片** —— 流程/架构/对比/时间线/组织关系/因果/路线图等结构化信息一律用画板承载。
4. **画板必须是原生可编辑画板** —— 点进去是可拖拽、可改节点文字的真画板。因此**禁止用 `<whiteboard type="svg">`**（那是把 SVG 当图嵌入，编辑性差、近似图片）。一律走「空白画板块 + `whiteboard +update`（Mermaid/PlantUML/DSL）」生成原生节点。
5. **默认严格保持文档顺序，确有必要才微调** —— 默认严格保持源文档的章节/段落原始先后顺序，画板就地插在它所解释的内容旁边；图片转出的内容也优先放回该图在正文里的原位置。这是默认行为,不要为了"看起来更顺"就随意重排。**例外**:当某处顺序明显妨碍逻辑/归类(例如同一主题被无关段落割裂、或几段本属同一组却散落各处)，可以适当调整顺序让它更清晰——但必须是你判断**确有必要**，且在 **Step 6 交付说明里逐条告知:哪里调了、为什么调**，让用户能复核。拿不准要不要调时，倾向保持原序。
6. **内容完整保真，不提炼、不删减（仅两类例外）** —— 新文档**正文的文字**默认必须包含源文档的**全部原文**，逐字保留，**不得概括、不得精简、不得改写任何文字**。所谓"整理"仅限于：排版/层级/格式优化，以及**新增**画板可视化。**允许删除的只有两类**:① **图片**——按铁律 11 转成它自身的提取内容、就地替换，最终文档不再保留原图（详见 Step 0）;② **近乎重复的内容**——当同一句话/同一条信息在文中出现 2 次以上、且措辞基本一致(达到 1 句以上的重复量)时，保留信息最完整的一处、删掉其余重复，免得读者反复读到同样的话。**判定从严**:只有确信是同一信息的重复才删;两段只是话题相近、措辞不同、或各自带有不同细节时**不算重复，一律保留**;拿不准是否真重复，就**保留并在 Step 6 交付说明里标注"疑似重复，已保留待你确认"**。除这两类外，正文不增不减、不改写文字。
7. **开头新增"导读区"** —— 在文档最前面（正文之前）新增一个开头总结区：①**文字归类总结**（把全文要点分类归纳）；②一张**统揽全局的原生可编辑画板**（总览/思维导图/分类结构）。这是**额外新增**的内容，与铁律 6 不冲突——正文原文依旧一字不动。
8. **整体条理清晰、逻辑分明** —— 全文读起来层次清楚、重点突出。这主要通过**排版手段**实现（标题分级、列表、加粗、callout 高亮、表格归位、留白等）。**调序、去重不算"排版"**:确需调整顺序或删除重复时，走铁律 5 / 6 的例外口子并在交付说明里讲清楚，不要借"让它更清晰"之名随意重排或删改原文。即默认"看起来更清晰"靠格式，不靠重排或删减。**emoji 要节制**：只在顶层章节标题（H1）点缀一个，不要给每个子标题（H2）都加 emoji，满屏 emoji 反而显乱、破坏层次。
9. **呈现形式优先级：可视化 > 表格 > 纯文字 ＋ 严禁大段文字** —— 同一块信息，**能画板可视化的优先画板**；不适合画板但有结构的，**优先用表格**承载；纯文字段落是最后选择。**多用表格**：并列项、对比、维度拆解、字段/术语说明、流程要点、Prompt 示例等一律转成 `<table>`，不要堆叠成连续段落。**严格避免出现大段纯文字**——任何超过约 3 行、内部含并列/对比/分点结构的密集段落，都要拆成表格或列表（大段文字看着累，是排版第一要消除的对象）。表格统一加表头底色（如 `<th background-color="light-blue">`）提升可读性。表格只是把原文重新排布、不增删改字（与铁律 6 不冲突）。
10. **重点强调分三级（由强到弱，按重要度选用）** ——
    - ① **最重点的句子 → 高亮块**：用 `<callout>` 把整句/整段框起来突出（最高优先级）。
    - ② **次重点 → 黄色背景**：`<span background-color="light-yellow">关键词句</span>`。
    - ③ **一般重点 → 红色加粗**：`<b><span text-color="red">关键词</span></b>`。
    三级均为**纯排版强调**，只给已有原文上色/加框，**不增删、不改写文字**。节制使用、拉开层次，不要每页都是强调反而没了重点。
11. **图片一律先读懂再转成原生内容，最终不留静态图** —— 源文档正文里嵌的图片（多为课件截图），必须**先用视觉读懂整张图**，再按图片类型转成对应的**原生可编辑内容**，放回原图位置（替换原图）。最终文档里**没有一张静态图**。这条与铁律 3「绝不生成静态图片」一脉相承，是它在"输入侧图片"上的延伸。具体转换规则和操作见 **Step 0**。
12. **省 token 纪律（和正确性同等重要）** —— 本工作流是 token 重耗任务，浪费多半出在两处：重复读图、为"看一眼"读高清图。硬约束：图片**一次性批量下载、每张只读一次、绝不重读**；读取失败先缩图、不对同一张反复重试；校验画板**优先文本核验（`--output_as code`），不为看一眼就导出读高清预览图**。详细操作见 Step 0 ② 与 Step 5 的 💸 提示，以及顶部「成本提示」。

## 工作流

```
源文档 URL ─► docs +fetch ──► 原始内容（全文 + 嵌入图片）
                                  │
                                  ▼
   【Step 0 图片预处理】逐张下载嵌入图 ─► 视觉读懂 ─► 按图型转成原生内容
                         （课件=知识点→案例文字 / 流程图=画板 / 表格截图=表格 /
                          代码截图=代码块 / 照片=文字描述）─► 放回原图位置，替换原图
                                  │
                                  ▼
            排版优化（默认不删/不减/不重排;去重与调序见铁律 5/6 例外） + 规划可视化点
                                  │
                                  ▼
   docs +create 新文档 = 【开头导读区：文字归类总结 + 总览画板】
                          + 【正文：原文全文 + 图片已转为文字/表格/代码/画板，顺序默认不变】
                                  │
                                  ▼
            就地插入原生可编辑画板（开头总览 + 正文各处）
```

### Step 0：图片预处理（把嵌入图转成原生内容）

源文档正文里若嵌了图片（多为课件截图），在正式整理之前先把它们处理掉，所以叫 Step 0。**实际执行顺序**:先做一次 Step 1 的 `docs +fetch` 拿到正文与图片清单 → 回到本 Step 0 把每张图转成原生内容 → 再继续 Step 2 起的整理流程。**核心思路:先用视觉读懂整张图，再按图片类型转成对应的原生可编辑内容，放回原图原位、替换原图——最终文档不留任何静态图。**

（源文档若不含任何图片，跳过本 Step 0，直接走 Step 1→Step 2。）

**① 识别并下载嵌入图片**

`docs +fetch` 的结果里，图片会以图片 block（含 image token / block_id）出现。**一次性批量下载全部图片**到本地临时目录(在一条命令或一批并行命令里下完,别一张一张来回),用 lark-doc 的图片/素材下载能力（先读 [`../lark-doc/SKILL.md`](../lark-doc/SKILL.md) 里"下载文档图片/附件/素材"一节，确认命令与所需 token）。记下每张图对应的 **block_id / 位置**，后面要按原位放回。

**② 视觉读懂每张图（先理解，再提取）**

对下载下来的每张图用 **Read 工具**（视觉）看懂它讲什么——这一步是"提取前先了解内容"，不要跳过直接抠字。读懂后判断它属于下表哪种类型，再决定输出形式。

> **💸 省 token(必看)**:**每张图只读一次**,看懂就一次性把要点/文字都记下来,**绝不重复读同一张图**。可以**一批多张一起读**(放进同一轮)以减少往返。若 Read 因尺寸过大失败(单图建议 ≤2000px),**先把图缩小再读,不要对同一张反复重试**(反复失败=纯浪费额度)。

**③ 按图片类型分流（所有图都转成原生可编辑内容，绝不留静态图）**

| 图片类型 | 输出形式 |
|---|---|
| 课件式「知识点 + 案例」 | 「知识点 → 案例」**文字**（知识点在前，对应案例紧随其后）|
| 流程图 / 架构图 / 关系图 | 原生**画板**（Mermaid，按 Step 5 的方式写入）|
| 表格截图 | 原生**表格** `<table>`（按铁律 9 加表头底色）|
| 代码截图 | 原生**代码块**（保留语言与缩进）|
| 照片 / 无结构图 | 忠实**文字描述**（说清图里有什么）|

- 判断类型前**必须先看懂整张图**；看不准就偏向"忠实转写其文字与结构"，不要臆造。
- 一张课件图里有多组「知识点+案例」→ 按原图顺序重复多组。
- 一张图混合多种结构（如既讲知识点又配流程图）→ 组合输出（知识点文字 + 一张流程画板）。
- **忠实为先**：只转述图里真实有的内容，不补充图里没有的信息、不删图里讲到的要点。

**课件图提取模板**（每张课件图按此输出，多组就重复）:

```
**知识点**
（忠实转述这张课件讲的知识点）

**案例**
（对应的案例）
```

> 不要给提取内容加「来源:图N」之类的出处标注——直接融入正文，读起来更顺。

**④ 放回原位、替换原图**

把每张图转出的内容，**放回它在正文里的原位置**（对应那张图原来所在的地方），替换掉原图。**顺序严格不动**（铁律 5）：不前移、不后移、不影响周围任何文字的位置。图片以外的正文文字仍逐字保真（铁律 6）。

处理完所有图后，正文里已经没有图片 block、全是原生内容，再继续 Step 2 起的整理流程。

### Step 1：读取源文档（只读）

```bash
lark-cli docs +fetch --api-version v2 --doc "<源文档 URL 或 token>"
```

- 先读 [`../lark-doc/references/lark-doc-fetch.md`](../lark-doc/references/lark-doc-fetch.md) 选好 `--scope` / `--detail`。
- 内容含嵌入的 `<sheet>` / `<bitable>` 等标签时，按 lark-doc 的规则提取 token 切到对应 skill 下钻读取，不能只呈现标签。
- **不要对源文档执行任何写操作。**

### Step 2：整理内容（全文保真，仅优化排版）

- **逐字保留源文档全部原文**：不概括、不精简、不改写。"整理"只允许做排版/层级/格式优化，以及新增画板。（图片已在 Step 0 转成原生内容并放回原位，此处把它们当正文一同保真处理，不再回退成图。）
- **去重例外**（铁律 6）：仅当出现 1 句以上、措辞基本一致的重复内容时，保留一处、删掉其余重复；判定从严，拿不准就保留并在 Step 6 标注。话题相近但措辞/细节不同的不算重复。
- **呈现形式按优先级选**（铁律 9）：可视化（画板）> 表格 > 纯文字。**优先把成块的并列/对比/维度信息转成 `<table>`**，少用连续纯文字段落。
- **重点按三级强调**（铁律 10）：最重点句子用 `<callout>` 高亮块；次重点用 `<span background-color="light-yellow">` 黄底；一般重点用 `<b><span text-color="red">` 红色加粗。
- 校对：整理后做一次比对，确保信息量与原文一致，没有内容遗漏或被压缩。
- 同步标记哪些部分**适合画板可视化**（这是在原文之外**新增**画板，不替换原文）：流程步骤、系统/模块架构、对比表、时间线/里程碑、组织/层级关系、因果链、路线图、关键指标关系。
- 纯叙述性、无结构的内容不强行画图。

### Step 3：撰写开头"导读区"（文字归类总结）

在正文之前新增一段开头总结（**纯新增，不动正文原文**）：

- **文字归类总结**：把全文要点按主题/类别归纳成几组（用小标题 + 列表），让读者开头就能抓到全貌。
- 预留一处**总览画板**位置（思维导图 / 分类结构 / 全局流程），具体画板在 Step 5 用原生方式写入。

### Step 4：另存为新文档（导读区 + 原文全文）

```bash
# XML（默认，可用 callout/grid/checkbox 等富 block）
lark-cli docs +create --api-version v2 --content '<title>【整理】<原标题></title>...导读区... ...原文全文...'
# 或 Markdown
lark-cli docs +create --api-version v2 --doc-format markdown --content $'<title>【整理】<原标题></title>\n导读区\n原文全文'
```

- 文档结构 = **开头导读区（Step 3 的归类总结）** + **正文（源文档全文，逐字、顺序不变）**。
- 从零创建先读 [`../lark-doc/references/style/lark-doc-create-workflow.md`](../lark-doc/references/style/lark-doc-create-workflow.md) 和 [`../lark-doc/references/lark-doc-xml.md`](../lark-doc/references/lark-doc-xml.md)。
- 记录返回的**新文档 token / URL** 及各 block_id，后续画板都写入这份新文档。
- 默认落在个人云空间，不进 wiki。
- **超长文档实践**：`docs +create` 只建「标题 + 导读区 + 总览画板」骨架，正文按章节分多次 `docs +update --command append` 写入；`--content` 用 `@相对路径文件` 从文件读取（须在当前工作目录内），避免命令行超长与转义问题。

### Step 5：插入原生可编辑画板（开头总览 + 正文各处）

> 关键规则（来自 lark-doc）：复杂画板可隔离到 SubAgent；但 Mermaid 图主 Agent 可直接写入。
> 本工作流**统一只走原生画板路径**（不论图简单还是复杂），保证用户点进去是可编辑状态。**禁用 `<whiteboard type="svg">`。**

画板分两类，都按下面同一套流程插入：
- **开头总览画板**（必做）：放在导读区，呈现全文分类结构 / 思维导图 / 全局流程。
- **正文各处画板**：每个 Step 2 标记的可视化点，就地紧挨它所解释的内容。

两种等价的写入方式（任选，实测均生成原生可编辑节点）：

- **方式 A · 内联 Mermaid（推荐，省一步）**：直接在 `docs +create` / `docs +update` 的内容里写
  ```xml
  <whiteboard type="mermaid">
  flowchart LR
    A[步骤一] --> B[步骤二] --> C[步骤三]
  </whiteboard>
  ```
  注意：内联在文档 XML 里时，Mermaid 文本中的 `&` 要转义为 `&amp;`，且**不要出现 `<`**（如 `<br/>`）；节点标签用引号包裹 `["含 / 和（）的文字"]` 更稳。

- **方式 B · 空白块 + 回填**：先插入空白画板拿 board_token，再用 `.mmd` 文件回填（可用 `<br/>`、特殊字符，无 XML 转义负担）。
  ```bash
  # 1) 插入空白画板（保持顺序用 block_insert_after --block-id <锚点>）
  lark-cli docs +update --api-version v2 --doc "<新文档 token>" --command append \
    --content '<whiteboard type="blank"></whiteboard>'
  # 从响应 data.new_blocks[] 中 block_type=="whiteboard" 的那条取 block_token
  # 2) 回填 Mermaid
  lark-cli whiteboard +update --whiteboard-token "<board_token>" \
    --input_format mermaid --source @./diagram.mmd --overwrite --as user
  ```

写入后做一次自检,确认 Mermaid 渲染成原生节点。**优先用文本核验,不要读预览大图**(读高分辨率 PNG 又贵又常因尺寸超限失败):

```bash
# 推荐:文本核验——导出节点文本,grep 关键节点名是否都在(便宜、稳)
lark-cli whiteboard +query --whiteboard-token "<board_token>" --output_as code --as user
```

只要关键节点文字都出现了,就说明 Mermaid 解析成功、是原生可编辑节点。**仅当确实需要肉眼看版面时**才导出图片,且导出后**先缩到 ≤1500px 再读、只读一次**:

```bash
# 可选:确需看版面时才用,读前务必缩图
lark-cli whiteboard +query --whiteboard-token "<board_token>" --output_as image --output ./preview.png --as user
```

- 飞书 Mermaid 实测支持 `mindmap`（思维导图）、`flowchart`/`graph`、`sequenceDiagram`、`classDiagram`、`pie`、`gantt`。架构/分层/分类用 `mindmap` 或 `flowchart TD`；流程用 `flowchart LR`；对比用 `subgraph`。
- 每个可视化点重复一次。**始终用原生画板，绝不用 `media-insert` 贴图、绝不用 SVG 嵌入代替。**

### Step 6：交付

- 给出**新文档**的 URL（以及每个画板所在位置）。
- 一句话说明：原文档未改动，已另存为新文档，正文原文逐字保留，开头新增归类总结 + 总览画板，正文 X 处用原生可编辑画板可视化。
- **若动用了铁律 5 / 6 的例外，必须逐条交代**：调整过顺序 → 列出"哪段移到哪、为什么"；删过重复 → 列出"删了哪几处重复、保留在何处"；有疑似重复但没删 → 标注出来请用户确认。没动这两项就不必提。
